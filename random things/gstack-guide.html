一個指令，全部完成

打開 Claude Code，貼上以下指令。剩下的交給 Claude。

git clone https://github.com/garrytan/gstack.git ~/.claude/skills/gstack && cd ~/.claude/skills/gstack && ./setup

需求：Claude Code、Git、Bun v1.0+。不會動到你的 PATH，也不會在背景執行任何程序。

加入你的專案讓隊友也能使用（選用）：

cp -Rf ~/.claude/skills/gstack .claude/skills/gstack && rm -rf .claude/skills/gstack/.git && cd .claude/skills/gstack && ./setup

真實檔案直接 commit 進 repo，非 submodule。git clone 即可使用，零設定。

你的虛擬工程團隊

每個技能都是一位專家。按順序使用效果最佳。

階段 1 — 思考與策略
/office-hours
YC 合夥人
從這裡開始。在寫第一行程式碼之前。

六個強制提問重新定義你的產品想法。挑戰你的思維框架、質疑前提假設，並產生多種實作方案。自動生成設計文件，供所有後續技能使用。

新功能 新專案 產品腦力激盪 「我有個想法」
階段 2 — 規劃與設計
/plan-ceo-review
執行長 / 創辦人
重新思考問題。找出隱藏其中的十星產品。

四種模式：擴展範疇（大膽想像）、選擇性擴展（精選擴充）、守住範疇（最高嚴謹度）、縮減範疇（回歸核心）。挑戰你的假設，找出需求背後真正的產品。

在 /office-hours 之後 「這夠有野心嗎？」 策略審查
/plan-eng-review
工程主管
在動手開發前鎖定架構。

ASCII 資料流程圖、狀態機、錯誤路徑、測試矩陣、失敗模式與安全疑慮。強制將隱藏假設攤在陽光下。在程式碼階段修正遠比事後便宜得多。

開始寫程式前 架構審查 plan-ceo-review 之後
/plan-design-review
資深設計師
在畫出第一個像素之前的設計審查。

對每個設計面向評分 0–10，說明滿分10分的標準，並直接修改計劃以達標。偵測 AI 生成的糟糕視覺模式。互動式審查——每個設計決策都會先詢問你。

UI/UX 規劃 前端開發前 設計評審
/design-consultation
設計夥伴
從零建立完整的設計系統。

研究視覺競品、提出美學方向、字型、色彩、間距與動態設計。生成字型與色彩預覽頁面。建立 DESIGN.md 作為專案設計的唯一真相來源。

尚無設計系統的新專案 「建立品牌規範」
/autoplan
審查流水線
一個指令，自動完成全套計劃審查。

自動依序執行 CEO → 設計 → 工程審查，並依照內建決策原則自動處理非判斷性決策。只將真正的主觀取捨（方向選擇、範疇平衡）提交給你決定。取代15–30個互動問題。

想讓審查自動完成時 「自動審查」
階段 3 — 審查與除錯
/review
資深工程師
找出能通過 CI 卻會在正式環境爆炸的 bug。

分析你與主分支的 diff。自動修復明顯問題。標記 SQL 安全、LLM 信任邊界違規、條件性副作用與完整性缺口。非顯而易見的部分會先詢問你。

每個 PR 之前 實作完成後 「程式碼審查」
/investigate
除錯專家
系統性根因除錯。不猜測，只調查。

四個階段：調查、分析、假設、實作。鐵則：沒有根因就不修復。追蹤資料流、驗證假設。若三次修復嘗試均失敗，停止並回報發現，而非無限循環。

「為什麼壞了？」 非預期行為 正式環境 bug
/design-review
會寫程式的設計師
在線上網站進行視覺 QA，然後直接修復。

與 /plan-design-review 相同的審查方法，但作用於你運行中的應用程式。找出 AI 糟糕視覺模式、間距不一致、層次結構問題。每個問題都用獨立 commit 修復，並附上修復前後截圖。

線上網站優化 「審查設計」 視覺不一致
/codex
第二意見
來自 OpenAI Codex CLI 的獨立審查。

三種模式：review（通過/不通過的 diff 分析）、challenge（對抗性模式，嘗試攻破你的程式碼）、consult（持續對話式問答）。當 /review 與 /codex 都跑過後，會進行跨模型交叉分析。

「第二意見」 高風險程式碼 對抗性測試
/cso
資訊安全長
完整的 OWASP Top 10 + STRIDE 威脅模型稽核。

零噪音設計：17個誤報排除規則、8/10以上信心門檻、獨立發現驗證。每個發現都附上具體的攻擊情境。兩種模式：日常（零噪音）與全面（每月深度掃描）。跨次稽核趨勢追蹤。

上線前 安全稽核 處理用戶資料時
階段 4 — 測試與 QA
/qa
QA 主管
用真實瀏覽器測試、找 bug、修 bug、再驗證。

開啟 Chromium，點擊你的真實使用流程，找出 bug，用獨立 commit 修復，並重新驗證每個修復。自動為每個修復生成回歸測試。三個層級：快速（僅嚴重問題）、標準、全面。產出修復前後的健康評分。

/qa https://staging.url 「測試這個功能」 發布前 QA
/qa-only
QA 報告員
與 /qa 相同，但只產報告——不動程式碼。

產出結構化 bug 報告，包含健康評分、截圖與重現步驟。當你想先了解哪些東西壞了再決定修什麼時使用，或需要乾淨稽核紀錄而不想改動程式碼時。

「只要回報 bug」 稽核紀錄 在 /qa 之前
/browse
QA 工程師
真實 Chromium 瀏覽器，每個指令約100毫秒。

導航頁面、點擊元素、驗證狀態、操作前後 diff、截圖標注、測試響應式佈局、表單、上傳、對話框。/qa 內部使用此工具——需要細粒度瀏覽器控制時直接呼叫。

所有網頁測試 「在瀏覽器開啟」 截圖驗證
/setup-browser-cookies
Session 管理員
匯入真實瀏覽器的 cookie，測試需登入的頁面。

開啟互動式選擇 UI，讓你從 Chrome、Arc、Brave 或 Edge 選擇要匯入哪些網域的 cookie 到無頭瀏覽器 session 中。測試需要登入的頁面前先執行此指令。

測試需登入頁面前 「登入網站」
/benchmark
效能工程師
建立基準並在每個 PR 比較頁面效能。

測量頁面載入時間、Core Web Vitals（LCP、CLS、FID）與資源大小。建立基準後，每個 PR 都進行前後比較。追蹤效能趨勢，在用戶感受到問題前發現退化。

「效能」 頁面速度 Bundle 大小 PR 前後比較
階段 5 — 發布與監控
/ship
發布工程師
同步主分支、跑測試、升版本、建立 PR。

偵測並合併基礎分支、執行測試套件、稽核覆蓋率、升 VERSION、更新 CHANGELOG、commit、push 並開啟 PR。若你尚無測試框架，會自動初始化一個。在 /review 之後執行。

程式碼準備好時 「建立 PR」 「部署」
/land-and-deploy
發布工程師
合併 PR、等待 CI、驗證正式環境健康狀態。

從「PR 已核准」到「正式環境已驗證」，一個指令搞定。合併 PR、等待 CI 與部署流水線完成，再執行 canary 檢查驗證正式環境是否健康。接在 /ship 之後使用。

PR 核准後 「上線」 「合併並部署」
/canary
SRE
部署後監控正式環境有無錯誤。

部署後的監控循環。使用真實瀏覽器檢查 console 錯誤、效能退化與頁面失敗。定期截圖，與部署前基準比較，發現異常時發出警報。

「監控部署」 部署後觀察 正式環境健康
/setup-deploy
DevOps
/land-and-deploy 的一次性部署設定。

偵測你的部署平台（Fly.io、Render、Vercel、Netlify、Heroku、GitHub Actions、自訂）、正式環境 URL、健康檢查端點與部署狀態指令。將設定寫入 CLAUDE.md。每個專案只需執行一次。

首次設定 「設定部署」
階段 6 — 文件與回顧
/document-release
技術寫作者
更新所有文件以符合你剛發布的內容。

讀取所有專案文件，與 diff 交叉比對，更新 README / ARCHITECTURE / CONTRIBUTING / CLAUDE.md 以反映已發布的內容。優化 CHANGELOG 語氣，清理 TODO，可選擇升版本號。自動發現過期的 README。

發布後 「更新文件」 PR 合併後
/retro
工程主管
具團隊感知的每週回顧，含趨勢追蹤。

分析 commit 歷史、工作模式與程式碼品質指標。每位成員的貢獻分解，含讚美與成長建議。執行 /retro global 可跨所有專案與 AI 工具（Claude Code、Codex、Gemini）彙總。持久化歷史記錄。

「每週回顧」 Sprint 結束 團隊 check-in
強力工具 — 安全與實用程式
/careful
安全護欄

在執行破壞性指令前發出警告：rm -rf、DROP TABLE、force-push、git reset --hard、kubectl delete。每個警告都可以覆蓋。操作正式環境或除錯線上系統時使用。

/freeze
編輯鎖定

將檔案編輯限制在特定目錄內。在除錯時防止意外修改目標範圍外的程式碼。想聚焦在單一模組而不小心「修好」無關程式碼時使用。

/guard
完整安全模式

/careful + /freeze 合而為一。正式環境工作或共享環境的最高安全防護。想同時擁有破壞性警告與目錄範圍限制時使用。

/unfreeze
解除鎖定

移除 /freeze 設定的邊界，無需結束 session。完成範圍除錯後，想允許對整個 repo 進行編輯時使用。

/gstack-upgrade
自動更新

升級 gstack 至最新版本。自動偵測全域或專案內安裝，執行升級並顯示更新日誌。當技能感覺過時或出現「invalid SKILL.md」錯誤時執行。

/codex
跨模型審查

也可作為獨立強力工具使用。三種模式：review、challenge、consult。當 /review 與 /codex 都執行過後，會進行交叉比對發現，提供更豐富的分析。
